# TL;DR

- AI に正確な仕事を任せるためには、初手として **「AI が働きやすい形でプロジェクトの情報を整える」** ことが重要
- Claude Code の Skills 機構を使って `.claude/skills/<project-name>/` 以下に **SSOT（Single Source of Truth）** として構築した
- やったことは **3 ステップ**：
    1. 散らばっているリソース（PDF・スライド・スプレッドシート・スクショ・ヒアリングメモ etc.）を一箇所にかき集める
    2. プランモードで「SSOT として整理してほしい」と AI に依頼する
    3. AI が整理整頓した Markdown 群が出来上がる。元リソースは用済み（gitignore）
- これは **Harness Engineering** の実践でもあり、**PM の新しい仕事の形** だと思っている

---

> **Prompting is temporary. Structure is permanent.**

https://x.com/i/status/2029836408232497678

この構成はこちらのXのポストにインスパイアされています。あまりにも管理する情報やステークホルダーが多岐に渡りすぎて危機感を覚え、どうにか人間にもAIにも事故なく分かりやすい形で情報を一元管理したいと思っており、コンテキストエンジニアリングを念頭に構造化していた時でした。
> If you want Claude Code to feel like a senior engineer living inside your repo, your project needs structure.

これを体現したいと思いました。

# 背景：このプロジェクトについて

私は現在、**全社横断の評価プログラム**で Technical Program Manager を務めています。プログラム自体をゼロから立ち上げ、設計・運用の全体責任を持っています。プロジェクトの性質を抽象化して書くと以下のような特徴があります。

## 1. ステークホルダーが多岐に渡る全社横断プロジェクト

- 対象となる 40 以上の組織の責任者
- 広報・クリエイティブチーム
- 運営メンバー
- エグゼクティブ層

「誰が何を見ていて、誰が何の責任を持っているか」が最初から複雑。PM が「プロジェクトの全体像を把握している唯一の人」になりがちな構造です。

## 2. 評価スキームが複雑で、業務ドメイン知識化していた

14ある項目それぞれに 複数のLv定義があり、さらに「LvXX 以上は根拠の妥当性を別プロセスで検証」「特定条件を満たすとランク上限が変わる」「特別表彰枠がある」といった運用ルールが乗っています。

この**評価ロジックそのものが業務ドメイン知識**として厚くなっていました。プロジェクトの説明をするだけでも多くの説明が必要になるし、AI にタスクを振ろうにも「現段階での最新情報」や「プロジェクト概要、背景説明」のフェーズが毎回発生。

## 3. 誤った情報開示が許されない ＝「決定事項だけ」がコンテキストに乗る必要がある
 
SSOT 設計でもっとも気を遣ったのは、**議論過程の情報ではなく、最終的に決定した情報 "だけ" がコンテキストに乗っている状態** を作ることでした。
 
ノイズが多いと、どれだけ強いモデルを使っても正確性が保てません。

---

いくらドキュメントを整備したとて「全情報を頭に入れた PM がボトルネックになる」構造が見えていました。

そのためこれを逆手に取って **「AI に全部の情報を持たせて、PM は必要なアウトプットを必要なフォーマットで取り出せる役割に回る」** 設計にすることにしました。

この記事では、そのための**土台作り** — SSOT をリポジトリに構築する部分 — を書きます。実際にその上で何をしたか（評価判定、レポート生成、ダッシュボード等）は、後続の記事で書く予定です。

# はじめに：PM の仕事を「情報の交通整理」から剥がす

**プロジェクトに関する情報があらゆるところに散らばっている**問題をどうにか一元化したい（しかもAIが働きやすい形で）と思ったことが出発点です。

- 説明会のスライド（PDF, Google Slides）
- ヒアリング結果（議事録, 録音, スプレッドシート）
- 業務定義（Notion, Google Docs, Slides）
- 評価基準・ルール（全体共有用とエクスクルーシブバージョンと微妙に載ってる情報の異なるスプレッドシート）
- タスク（Notion, GitHub Issues）
- 関連 URL（社内ポータル、業界資料、認証つきサイト）

「組織◯◯の責任者をXXに変更したいです」「LvXX の観点◯◯の定義もう一回教えてください」「最新の対象組織のリストどこにありますか」

私はこの問題の解き方として、**プロジェクトの情報を全部 `.claude/skills/` に押し込んだ SSOT を作る**ことにしました。この記事はその作り方の話です。

# 用語の整理：Harness Engineering と SSOT

本題に入る前に、AI エンジニアリング用語の整理です。

```
Prompt Engineering    (2022-2024)   "What should I say?"
Context Engineering   (2024-2025)   "What should I show?"
Harness Engineering   (2026〜)      "How should it run?"
```

> **Agent = Model + Harness**

モデルはエンジン、ハーネスはそれを覆う車体。プロンプトもコンテキストもハーネスの中の一要素にすぎない、というのが現在の定説です（[OpenAI: Harness engineering](https://openai.com/index/harness-engineering/)）。

https://openai.com/index/harness-engineering/

そしてハーネスの設計で最も基本的なレイヤーが **「コンテキスト境界管理」** ＝ AI が見る情報を構造化して整理することです。SSOT（Single Source of Truth、唯一の信頼できる情報源）はこのレイヤーの実装パターンになります。

PM の仕事を Harness Engineering の言葉で言い換えると、こうなります：

> **PM = AI エージェントが働ける環境を設計する人**

「情報の交通整理係」から「ハーネス設計者」への転職です。

# Claude Code プロジェクトの「解剖図」と、本記事のスコープ

本題に入る前に、**Claude Code プロジェクトとしてよく設計されたリポジトリがどういう構造を持っているか** の全体像を一度示しておきます。冒頭で引用した元ポストが "Anatomy of a Claude Code Project" として 5 つの要素を挙げていて、これが俯瞰として非常にわかりやすいので拝借します。

| # | 要素 | 役割 | ひとことで言うと |
|---|---|---|---|
| 1 | `CLAUDE.md`（ルート） | Repo Memory | プロジェクトの**北極星**。WHY / 構成マップ / ルール・コマンドを**短く** |
| 2 | `.claude/skills/` | Reusable Expert Modes | **再利用可能なワークフロー** を Markdown として切り出す |
| 3 | `.claude/hooks/` | Guardrails | 「必ずやる」をフックで自動化（フォーマット、テスト、危険ディレクトリのブロック） |
| 4 | `docs/` | Progressive Context | アーキ図・ADR・Runbook 等の**参照先** |
| 5 | `src/<module>/CLAUDE.md` | Local Memory | 危険箇所（auth / billing / migration 等）の**ローカル注意書き** |

そして Claude が安定して動くためには**常に 4 つの情報**が必要と整理されています。

| 必要な情報 | 意味 | この 5 要素での置き場所 |
|---|---|---|
| **WHY** | システムが何をするか | ルート `CLAUDE.md` |
| **MAP** | 物がどこにあるか | ルート `CLAUDE.md` + `docs/` |
| **RULES** | 何が許可され、何が禁止か | `CLAUDE.md` + `.claude/hooks/` + ローカル `CLAUDE.md` |
| **WORKFLOWS** | 仕事をどう進めるか | `.claude/skills/` |

## この記事が扱う範囲

本記事が深掘りするのは、上記のうち **#2 の `.claude/skills/`** です。理由は以下：

- PM がプロジェクト情報を SSOT 化するときに**最も活用度が高い**のがここ
- ドメイン知識・業務ルール・意思決定の記録など、**「ワークフロー + 参照情報」の複合体** を置くのに向いている
- `CLAUDE.md` だけに詰めると肥大化して逆効果（元ポストも「長くなると重要な文脈を見落としはじめる」と指摘）なので、**分離した方が結果的に `CLAUDE.md` もクリーンに保てる**

他の 4 要素（`CLAUDE.md` / `hooks` / `docs` / Local `CLAUDE.md`）は必要に応じて併用していますが、**本記事では詳しく扱いません**。これらは別記事で書く予定です。

# なぜ Notion / MCP 連携ではなく、Claude Skills にしたか

「プロジェクト情報を SSOT 化したい」と聞くと、Notion とか Confluence に情報を集めて、MCP で AI に繋ぐ、という方法もあるかと思います。実際、それも有力な選択肢です。

でも私は最終的に **Claude Skills（`.claude/skills/` 配下の Markdown とスクリプト）** を選びました。理由は以下です。

## 1. ローカル志向 — リポジトリに入る・git でバージョン管理できる

`.claude/skills/` はただのローカルファイルです。リポジトリにコミットすれば：

- **PR ベースで変更できる**：「Lv3 の定義を変えたい」→ PR 立てる → レビュー → マージ、でルールが即日 AI の振る舞いに反映される
- **diff で変更履歴が追える**：Notion の版管理より git の履歴の方が圧倒的に強い
- **AI が勝手にコードの近くで読みに行く**：コードと判定ルールが同じリポジトリにあるので、AI がコード生成しながら同時にルールも参照できる

MCP + Notion にすると、「ルールが外部サービスに住んでいて、API 越しに AI が取りに行く」構造になります。これは便利な反面、**外部依存**が増えるし、書き換えと反映のサイクルも git ほどシームレスではない。

## 2. 軽量 — サーバーもトークンも認証も要らない

MCP サーバーを立てると以下が必要になります：

- MCP サーバープロセスの運用
- Notion API トークン管理
- 認証とエラーハンドリング
- レート制限への対応

Skills は **Markdown を置くだけ**。サーバーもトークンもない。Claude Code が勝手に読む。これが「立ち上げフェーズ」の PM にとってはありがたい。

実務でも、「プロジェクト初期で、とにかく早く土台を組みたい」フェーズにおいて、**MCP 構築のコストは重すぎる**と感じました。プロジェクトが成熟して、複数メンバーが並行で AI と連携するフェーズになったら、また別の選択肢として出てくるかもしれない。

## 3. 運営が実質 1 人だった（= Claude Code との相性が最高）

本プロジェクトでは**手を動かす運営が実質 1 人**でした。PM 兼エンジニア、という体制です。ということは：

- ドキュメントの閲覧者 = 主に自分 + AI
- ドキュメントの編集者 = 主に自分 + AI
- 実行環境 = 自分のローカル

この構造において Claude Code は相性が良い。Skillは標準化された規格のためタスクの性質によってはCodexを使うこともしばしばありましたが、シームレスにコンテキストを共有できました。

# 完成形：`.claude/skills/<project>/` の構造

最終的に出来上がったディレクトリ構造はこうなります（プロジェクト名はダミー）。

```
.claude/skills/
└── my-project/
    ├── SKILL.md                # プロジェクトのマスタードキュメント
    └── references/
        ├── overview.md          # プロジェクトの目的・全体像
        ├── scheme.md            # 座組（誰がどの役割か、組織分担）
        ├── evaluation.md        # 評価基準・判定ルール
        ├── schedule.md          # スケジュール・マイルストーン
        ├── faq.md               # よくある質問・例外規則
        ├── data.json            # ドメインデータ
        └── helpers/             # スクリプト類
            └── score.js
```

ポイントは以下の 3 つです：

1. **`SKILL.md` がマスター**：YAML フロントマターで `name` と `description` を書くと、Claude Code は呼び出し時にディスクリプションだけを読み、必要な時に本文と references を読みに行く（遅延ロード）
2. **`references/` 配下に小分け**：1 つの巨大なファイルではなく、関心事ごとに分割。Markdown だけでなく **JSON / シェルスクリプト / 何でも置ける**
3. **「ジップファイルみたいに一塊」**：このディレクトリ全体が 1 つのスキルパッケージ

`SKILL.md` の冒頭はこんな感じ：

```markdown
---
name: my-project
description: "プロジェクトXの全情報を統合管理するスキル。座組・評価基準・スケジュール・FAQ等を含む。「Xプロジェクト」「A 組」「評価」等のキーワードで使用。"
---

# プロジェクト X — マスタードキュメント

## プロジェクト概要

詳細は `references/overview.md` を参照。

## 座組

各役割と担当者は `references/scheme.md` を参照。

## 評価基準

判定ルールは `references/evaluation.md` を参照。
...
```

description にキーワードを詰めるのが命です。雑に書くと AI が呼んでくれません。

# どうやって AI にこの整備をしてもらったか

やったことはシンプルで、**材料を集めて、目的を伝えて、AI に整理させる**だけです。

私自身が最初から Markdown を全部手で書いたわけではありません。先にプロジェクトの素材を集めて、その上で Claude Code に「この情報を Skills として整理してほしい」と依頼しました。以下、そのときの進め方を 3 ステップで書きます。

---

# Step 1: リソースをかき集める

最初にやるのは、地味ですが**プロジェクトに関する素材を全部一箇所に集める**ことです。

```
my-project/
├── .claude/skills/        ← ここはまだ空
└── resources/             ← まずはここに全部突っ込む（gitignore）
    ├── overview/
    │   ├── kickoff_slides.pdf
    │   ├── intro_capture.png
    │   └── presentation_recording.txt
    ├── scheme/
    │   ├── members_list.csv
    │   ├── org_responsibility.xlsx
    │   └── hearing_results.csv
    └── evaluation/
        ├── criteria_industry_a.png    ← 業界事例のスクショ
        ├── criteria_industry_b.png
        └── evaluation_levels.xlsx
```

## 集めるもの

私の場合はこんなのを集めました：

- **キックオフ説明会のスライド**（PDF or PPT）
- **ヒアリングシートの結果**（スプシエクスポート）
- **業界事例ページのキャプチャ**（認証が面倒なサイトはスクショで OK）
- **評価レベル定義の Excel**
- **メンバーリスト・組織図**
- **過去の議事録**

> 認証が必要なサイト・複雑な動的ページは、Playwright MCP で取りに行くか、面倒なら**ブラウザでスクショして PNG として置く**。AI は画像も読めるので、これで十分。

## 重要な原則：**「まずは全部入れる」**

「これは必要かな？」と判断しない。判断するのは AI の仕事。**人間の仕事は集めることだけ**。

ただし、後述しますがこの `resources/` は **gitignore でコミットしません**。重いし、最終的に用済みになります。

## 大事な構造化のコツ：「プロジェクトの本質的な軸」でフォルダを切る

リソースを集める時、ただバラバラに突っ込むのではなく、**プロジェクトの本質的な軸**でフォルダを分けると後の Step 2 が劇的に楽になります。

私のプロジェクトの場合、本質的な軸は **「座組（scheme）」** と **「評価（evaluation）」** の 2 つでした。

- **座組（scheme）**: 誰がどの組織のリーダーか、組織をどう分けるか、責任者は誰か、管轄はどこか
- **評価（evaluation）**: 判定基準のレベル定義、業界事例、スコア算出方法

この 2 軸はプロジェクトを動かす上での「2 つの大きな仕事」と完全に一致しているので、後で AI に「`resources/scheme/` 配下を読んで座組のスキルを作って」「`resources/evaluation/` 配下を読んで評価のスキルを作って」と依頼するときに、**指示が綺麗に分割できる**。

PM 的な観点で言えば、**「自分のプロジェクトの本質的な分担軸は何か」を明文化するフェーズでもあります**。これ自体が PM の重要な思考整理です。

---

# Step 2: AI にスキルを作らせる（Planモード活用）

リソースが集まったら、Claude Code を起動してプランモードで以下のような目的を伝えます。

```text
このリポジトリで、Claude Skills を使って
プロジェクト X の SSOT を作りたい。

目的:
- 人間も AI も、このスキルだけを見れば仕事ができる状態にしたい
- 情報が複数の場所に散財しないようにしたい (Single Source of Truth)
- Context Engineering の観点で、コンテキストを 1 箇所に集約したい

材料:
- resources/ 配下にプロジェクトの素材が一通り入っている
  - resources/overview/ : プロジェクト全体像
  - resources/scheme/   : 座組情報
  - resources/evaluation/ : 評価基準
- これらを読んで、AI が判断・実行に使える形に整理してほしい

成果物:
- .claude/skills/my-project/SKILL.md (マスター)
- .claude/skills/my-project/references/*.md (関心事別に分割)
- 元リソースで重要な情報は references/ に Markdown として転記
- PDF・画像・Excel などの生データは references には含めない
```

## キーワード「SSOT」を入れるとちゃんと作ってくれる

ポイントとして、プロンプトに "SSOT" や "Context Engineering" のような固有ワードを入れることでAI がそれっぽく作ってくれます。逆にこれらの言葉を使わずに「整理して」とだけ言うと、雑な構造になりがち。

「SSOT」は Context Engineering / Harness Engineering とセットでよく出てくるキーワードです。AI も学習データに大量にこの組み合わせを見てるので、ヒントになります。

最終的に着地するまでには何度もターンを繰り返しながら作り上げます。
初期コストはそれなりにかかりますが、これを 1 回ちゃんとやれば、後の作業が圧倒的に楽になります。

---

# Step 3: 整理されたスキルが完成、元リソースは用済み

数回のイテレーションで、こうなります：

```
my-project/
├── .claude/skills/my-project/
│   ├── SKILL.md
│   └── references/
│       ├── overview.md       ← AI が PDF やスライドから抽出して書いた
│       ├── scheme.md          ← AI がメンバー CSV と組織図から書いた
│       ├── evaluation.md      ← AI が Excel と業界スクショから書いた
│       ├── schedule.md
│       └── faq.md
└── resources/                  ← gitignore（もう用済み）
```

## 元リソースは gitignore して、コミットしない

PDF・Excel・スクショ・録音 — これらは：

- **重い**（リポジトリが肥大化）
- **AI に優しくない**（PDF は構造化されてないし、毎回 OCR が必要）
- **同じ情報が何回も書いてある**（ノイズ）
- **抽出後は用済み**（Markdown に落とせばそれが本物）

なので、`.gitignore` に `resources/` を追加して**ローカルだけに置く**。コミットするのは整理済みの `.claude/skills/` 配下だけ。

## スキル化が完成したあとの AI とのコミュニケーション

### Before（スキルなし時代）
```text
プロジェクト X についてヘルプして欲しい。背景としては、
うちの会社では半年に1回〇〇という取り組みをやっていて、
○○組には A 社と B 社が所属していて、責任者は…

それで、Lv3 以上の組織を抽出して欲しいんだけど、Lv3 の定義は…
```

### After（スキルあり時代）
```text
Lv3 以上の組織一覧を出して。
```

背景説明・用語定義・組織情報・評価基準、全部スキル内にあるので AI が自分で読みに行きます。**指示が業務語彙だけで完結する**。

副次的な効果として：

- **問い合わせ工数の削減** 「XX組織の◯◯の情報は？」をデータベースから参照せずともAIに丸投げできる
- **意思決定の質が高まる** 意思決定に至るまでの、ファクトや根拠を、複雑なドメイン知識から的確な情報をAIを介して取得できることで、意思決定の質が高まる
- **属人化が解消する**：自分意外のメンバーもリポジトリを参照することでAIというインターフェースを通して同じ結論が導き出せる

---

# まとめ

PM の本質的な価値は「**プロジェクトの全体像を理解していて、どこに何があるか分かっている**」ことから **「AI エージェントが働ける環境を設計する」 = Harness Engineering** に移ってきたように思います。

AI時代のプロジェクトマネジメントにおいて肝となるのは「**自分の仕事を AI に渡せる形に再構築する**」ことだと考えています。

---

ここまでが**プロジェクトを AI と共に動かすための土台作り**です。

この上で、実際に AI に評価判定をさせたり、ダッシュボードなどのアプリケーションを構築させたり、レポートを書かせたり…という具体的なタスクが乗ります。この話は別記事で書く予定です。
