# TL;DR

- AI に仕事を任せるためには、その前に **「AI が働きやすい形でプロジェクトの情報を整理する」** 仕事が必要になる。
- 私はこれを Claude Code の Skills 機構を使って `.claude/skills/<project-name>/` 以下に **SSOT（Single Source of Truth）** として構築した。
- やったことは **3 ステップ**：
    1. 散らばっているリソース（PDF・スライド・スプレッドシート・スクショ・ヒアリングメモ etc.）を一箇所にかき集める
    2. プランモードで「SSOT として整理してほしい」と AI に依頼する
    3. AI が整理整頓した Markdown 群が出来上がる。元リソースは用済み（gitignore）
- これは 2026 年の AI エンジニアリングのトレンド **Harness Engineering** の実践でもあり、**PM の新しい仕事の形** だと思っています。

# 背景：このプロジェクトについて

本題の前に、どういうプロジェクトで実践した話なのかを書いておきます。

私は現在、ある**全社横断の評価プログラム**で Technical Program Manager を務めています。プログラム自体をゼロから立ち上げ、設計・運用の全体責任を持っています。プロジェクトの性質を抽象化して書くと以下のような特徴があります。

## 1. ステークホルダーが多岐に渡る全社横断プロジェクト

- エグゼクティブ層
- 対象となる 40 以上の組織の責任者
- 広報・クリエイティブチーム
- 各組織の現場メンバー

「誰が何を見ていて、誰が何の責任を持っているか」が最初から複雑。PM が「プロジェクトの全体像を把握している唯一の人」になりがちな典型的な構造です。

## 2. 3 次元データを扱う評価スキームの複雑さ

評価の構造は以下の 3 次元：

- **評価観点**: 14 項目
- **レベル**: 5 段階（Lv1 〜 Lv5）
- **対象組織**: 40 以上

単純計算で **14 × 5 × 40 = 2,800 セル**の判定結果を扱います。さらに各セルには「根拠資料の妥当性」の検証プロセスが紐づくので、実際の情報量はその何倍にもなります。

Excel 一枚じゃ扱えない。スプシ + Docs + スライド + ヒアリング結果 + 業界事例 + 過去議事録…が全部必要になる性質のプロジェクトです。

## 3. 評価スキームが複雑で、業務ドメイン知識化していた

14 項目それぞれに Lv1-5 の定義があり、さらに「Lv3 以上は根拠の妥当性を別プロセスで検証」「特定条件を満たすとランク上限が変わる」「特別表彰枠がある」といった運用ルールが乗っています。

この**評価ロジックそのものが業務ドメイン知識**として厚くなっていました。新しいメンバーをオンボードするだけで数時間の説明が必要になるし、AI にタスクを振ろうにも「まず背景から説明」のフェーズが毎回発生する。

---

こういう性質のプロジェクトを 1 人の PM が回すのは、**普通にやったら無理**です。「全情報を頭に入れた PM がボトルネックになる」構造が確定している。

ただ、これを逆手に取って **「AI に全部の情報を持たせて、PM は目的を指し示す役割に回る」** という設計にできれば、このスケール感が一気に扱えるようになる。これが私の設計の出発点でした。

この記事では、そのための**土台作り** — SSOT をリポジトリに構築する部分 — を書きます。実際にその上で何をしたか（評価判定、レポート生成、ダッシュボード等）は、後続の記事で書く予定です。

---

# はじめに：PM の仕事は「情報の交通整理」じゃなくなった

PM をやっていると、**プロジェクトに関する情報があらゆるところに散らばっている**問題に必ずぶつかります。

- 説明会のスライド（PDF, Google Slides）
- ヒアリング結果（議事録, 録音, スプレッドシート）
- 業務定義（Notion, Confluence, Google Docs）
- 評価基準・ルール（Excel, スプレッドシート, 社内Wiki）
- タスク（Linear, Jira, GitHub Issues）
- 関連 URL（社内ポータル、業界資料、認証つきサイト）

新メンバーがジョインするたびに「あの情報はあそこの…」を 30 分くらい喋る、というのは PM あるあるだと思います。

そして 2026 年の今、ここに新しい問題が乗ります：**AI エージェントにも同じことを毎回説明しなきゃいけない**。

「A 組の責任者って誰でしたっけ？」「Lv3 の定義もう一回教えてください」「対象組織のリストどこにありましたっけ？」 — これを毎セッション AI に教えるのは無理です。**情報が AI に届く形になっていない**から。

私はこの問題の解き方として、**プロジェクトの情報を全部 `.claude/skills/` に押し込んだ SSOT を作る**ことにしました。この記事はその作り方の話です。

# 用語の整理：Harness Engineering と SSOT

本題に入る前に、2026 年の AI エンジニアリング用語の現在地を一行で。

```
Prompt Engineering    (2022-2024)   "What should I say?"
Context Engineering   (2024-2025)   "What should I show?"
Harness Engineering   (2026〜)      "How should it run?"
```

> **Agent = Model + Harness**

モデルはエンジン、ハーネスはそれを覆う車体。プロンプトもコンテキストもハーネスの中の一要素にすぎない、というのが現在の定説です（[OpenAI: Harness engineering](https://openai.com/index/harness-engineering/)）。

そしてハーネスの設計で最も基本的なレイヤーが **「コンテキスト境界管理」** ＝ AI が見る情報を構造化して整理することです。SSOT（Single Source of Truth、唯一の信頼できる情報源）はこのレイヤーの実装パターンになります。

PM の仕事を Harness Engineering の言葉で言い換えると、こうなります：

> **PM = AI エージェントが働ける環境を設計する人**

「情報の交通整理係」から「ハーネス設計者」への転職です。

# Claude Code プロジェクトの「解剖図」と、本記事のスコープ

本題に入る前に、**Claude Code プロジェクトとしてよく設計されたリポジトリがどういう構造を持っているか** の全体像を一度示しておきます。冒頭で引用した元ポストが "Anatomy of a Claude Code Project" として 5 つの要素を挙げていて、これが俯瞰として非常にわかりやすいので拝借します。

| # | 要素 | 役割 | ひとことで言うと |
|---|---|---|---|
| 1 | `CLAUDE.md`（ルート） | Repo Memory | プロジェクトの**北極星**。WHY / 構成マップ / ルール・コマンドを**短く** |
| 2 | `.claude/skills/` | Reusable Expert Modes | **再利用可能なワークフロー** を Markdown として切り出す |
| 3 | `.claude/hooks/` | Guardrails | 「必ずやる」をフックで自動化（フォーマット、テスト、危険ディレクトリのブロック） |
| 4 | `docs/` | Progressive Context | アーキ図・ADR・Runbook 等の**参照先** |
| 5 | `src/<module>/CLAUDE.md` | Local Memory | 危険箇所（auth / billing / migration 等）の**ローカル注意書き** |

そして Claude が安定して動くためには**常に 4 つの情報**が必要と整理されています。

| 必要な情報 | 意味 | この 5 要素での置き場所 |
|---|---|---|
| **WHY** | システムが何をするか | ルート `CLAUDE.md` |
| **MAP** | 物がどこにあるか | ルート `CLAUDE.md` + `docs/` |
| **RULES** | 何が許可され、何が禁止か | `CLAUDE.md` + `.claude/hooks/` + ローカル `CLAUDE.md` |
| **WORKFLOWS** | 仕事をどう進めるか | `.claude/skills/` |

## この記事が扱う範囲

本記事が深掘りするのは、上記のうち **#2 の `.claude/skills/`** です。理由は以下：

- PM がプロジェクト情報を SSOT 化するときに**最も活用度が高い**のがここ
- ドメイン知識・業務ルール・意思決定の記録など、**「ワークフロー + 参照情報」の複合体** を置くのに向いている
- `CLAUDE.md` だけに詰めると肥大化して逆効果（元ポストも「長くなると重要な文脈を見落としはじめる」と指摘）なので、**分離した方が結果的に `CLAUDE.md` もクリーンに保てる**

他の 4 要素（`CLAUDE.md` / `hooks` / `docs` / Local `CLAUDE.md`）は必要に応じて併用していますが、**本記事では詳しく扱いません**。これらは別記事で書く予定です。

# なぜ Notion / MCP 連携ではなく、Claude Skills にしたか

「プロジェクト情報を SSOT 化したい」と聞くと、Notion とか Confluence に情報を集めて、MCP で AI に繋ぐ、という構成を思い浮かべる人も多いと思います。実際、それも有力な選択肢です。

でも私は最終的に **Claude Skills（`.claude/skills/` 配下の Markdown とスクリプト）** を選びました。理由は以下です。

## 1. ローカル志向 — リポジトリに入る・git で管理できる

`.claude/skills/` はただのローカルファイルです。リポジトリにコミットすれば：

- **PR ベースで変更できる**：「Lv3 の定義を変えたい」→ PR 立てる → レビュー → マージ、でルールが即日 AI の振る舞いに反映される
- **diff で変更履歴が追える**：Notion の版管理より git の履歴の方が圧倒的に強い
- **AI が勝手にコードの近くで読みに行く**：コードと判定ルールが同じリポジトリにあるので、AI がコード生成しながら同時にルールも参照できる

MCP + Notion にすると、「ルールが外部サービスに住んでいて、API 越しに AI が取りに行く」構造になります。これは便利な反面、**外部依存**が増えるし、書き換えと反映のサイクルも git ほどシームレスではない。

## 2. 軽量 — サーバーもトークンも認証も要らない

MCP サーバーを立てると以下が必要になります：

- MCP サーバープロセスの運用
- Notion API トークン管理
- 認証とエラーハンドリング
- レート制限への対応

Skills は **Markdown を置くだけ**。サーバーもトークンもない。Claude Code が勝手に読む。これが「立ち上げフェーズ」の PM にとっては本当にありがたい。

実務でも、「プロジェクト初期で、とにかく早く土台を組みたい」フェーズにおいて、**MCP 構築のコストは重すぎる**と感じました。プロジェクトが成熟して、複数メンバーが並行で AI と連携するフェーズになったら、また別の選択肢として出てくるかもしれない。

## 3. 運営が実質 1 人だった（= Claude Code との相性が最高）

これが決め手だったかもしれません。

このプロジェクトは**手を動かす運営が実質 1 人**でした。PM 兼エンジニア、という体制です。ということは：

- ドキュメントの閲覧者 = 主に自分 + AI
- ドキュメントの編集者 = 主に自分
- 実行環境 = 自分のローカル

この構造において Claude Code は完璧に相性が良い。**自分のエディタで Claude Code を開いて、`.claude/skills/` の Markdown をいじって、そのまま AI に作業させる** という一連のフローが途切れずに回る。

Notion を間に挟むと、「ブラウザで Notion を開く → 編集する → AI に共有する」という文脈のスイッチが毎回発生します。1 人運営では**この摩擦が地味にきつい**。

もし運営が 5 人チームで、エンジニアじゃない PM や広報も直接ドキュメントを編集する、みたいな体制なら、Notion + MCP の方が良い可能性は全然あります。**選択は運営体制次第**。

## 補足：MCP 連携との併用も可能

ちなみに「Skills vs MCP」は二者択一ではなく、**併用できます**。

- **Skills**: SSOT としての「静的な知識」（ルール・定義・テンプレート）
- **MCP**: 動的な外部データ取得（Linear のタスク、Drive のファイル etc.）

今回の構成では、評価対象データの取得部分だけ別途 MCP / API で繋ぎ、**判定ロジックの SSOT は Skills** に置く、という役割分担をしました。

# 完成形：`.claude/skills/<project>/` の構造

最終的に出来上がったディレクトリ構造はこうなります（プロジェクト名はダミー）。

```
.claude/skills/
└── my-project/
    ├── SKILL.md                # プロジェクトのマスタードキュメント
    └── references/
        ├── overview.md          # プロジェクトの目的・全体像
        ├── scheme.md            # 座組（誰がどの役割か、組織分担）
        ├── evaluation.md        # 評価基準・判定ルール
        ├── schedule.md          # スケジュール・マイルストーン
        ├── faq.md               # よくある質問・例外規則
        ├── data.json            # ドメインデータ
        └── helpers/             # スクリプト類
            └── score.js
```

ポイントは以下の 3 つです：

1. **`SKILL.md` がマスター**：YAML フロントマターで `name` と `description` を書くと、Claude Code は呼び出し時にディスクリプションだけを読み、必要な時に本文と references を読みに行く（遅延ロード）
2. **`references/` 配下に小分け**：1 つの巨大なファイルではなく、関心事ごとに分割。Markdown だけでなく **JSON / シェルスクリプト / 何でも置ける**
3. **「ジップファイルみたいに一塊」**：このディレクトリ全体が 1 つのスキルパッケージ

`SKILL.md` の冒頭はこんな感じ：

```markdown
---
name: my-project
description: "プロジェクトXの全情報を統合管理するスキル。座組・評価基準・スケジュール・FAQ等を含む。「Xプロジェクト」「A 組」「評価」等のキーワードで使用。"
---

# プロジェクト X — マスタードキュメント

## プロジェクト概要

詳細は `references/overview.md` を参照。

## 座組

各役割と担当者は `references/scheme.md` を参照。

## 評価基準

判定ルールは `references/evaluation.md` を参照。
...
```

description にキーワードを詰めるのが命です。雑に書くと AI が呼んでくれません。

# 「いやこんな整理されたもの最初から書けるわけないでしょ」

って思いますよね。私もそう思います。

**書いたのは私じゃなくて AI です**。

私がやったのは、**目的の言語化 + リソースのかき集め** だけ。AI が整理整頓してくれます。具体的な手順を 3 ステップで書きます。

---

# Step 1: リソースをかき集める（ここが全て）

最初にやるのは、地味ですが**プロジェクトに関する素材を全部一箇所に集める**ことです。

```
my-project/
├── .claude/skills/        ← ここはまだ空
└── resources/             ← まずはここに全部突っ込む（gitignore）
    ├── overview/
    │   ├── kickoff_slides.pdf
    │   ├── intro_capture.png
    │   └── presentation_recording.txt
    ├── scheme/
    │   ├── members_list.csv
    │   ├── org_responsibility.xlsx
    │   └── hearing_results.csv
    └── evaluation/
        ├── criteria_industry_a.png    ← 業界事例のスクショ
        ├── criteria_industry_b.png
        └── evaluation_levels.xlsx
```

## 集めるもの

私の場合はこんなのを集めました：

- **キックオフ説明会のスライド**（PDF or PPT）
- **ヒアリングシートの結果**（スプシエクスポート）
- **業界事例ページのキャプチャ**（認証が面倒なサイトはスクショで OK）
- **評価レベル定義の Excel**
- **メンバーリスト・組織図**
- **過去の議事録**

> 認証が必要なサイト・複雑な動的ページは、Playwright MCP で取りに行くか、面倒なら**ブラウザでスクショして PNG として置く**。AI は画像も読めるので、これで十分。

## 重要な原則：**「ノイズになるものこそ、まずは全部入れる」**

「これは必要かな？」と判断しない。判断するのは AI の仕事。**人間の仕事は集めることだけ**。

ただし、後述しますがこの `resources/` は **gitignore でコミットしません**。重いし、最終的に用済みになります。

## 大事な構造化のコツ：「プロジェクトの本質的な軸」でフォルダを切る

リソースを集める時、ただバラバラに突っ込むのではなく、**プロジェクトの本質的な軸**でフォルダを分けると後の Step 2 が劇的に楽になります。

私のプロジェクトの場合、本質的な軸は **「座組（scheme）」** と **「評価（evaluation）」** の 2 つでした。

- **座組（scheme）**: 誰がどの組織のリーダーか、組織をどう分けるか、責任者は誰か、管轄はどこか
- **評価（evaluation）**: 判定基準のレベル定義、業界事例、スコア算出方法

この 2 軸はプロジェクトを動かす上での「2 つの大きな仕事」と完全に一致しているので、後で AI に「`resources/scheme/` 配下を読んで座組のスキルを作って」「`resources/evaluation/` 配下を読んで評価のスキルを作って」と依頼するときに、**指示が綺麗に分割できる**。

PM 的な観点で言えば、**「自分のプロジェクトの本質的な分担軸は何か」を明文化するフェーズでもあります**。これ自体が PM の重要な思考整理です。

# Step 2: AI にスキルを作らせる（プランモード活用）

リソースが集まったら、Claude Code を起動してプランモードで以下のような目的を伝えます。

```text
このリポジトリで、Claude Skills を使って
プロジェクト X の SSOT を作りたい。

目的:
- 人間も AI も、このスキルだけを見れば仕事ができる状態にしたい
- 情報が複数の場所に散財しないようにしたい (Single Source of Truth)
- Context Engineering の観点で、コンテキストを 1 箇所に集約したい

材料:
- resources/ 配下にプロジェクトの素材が一通り入っている
  - resources/overview/ : プロジェクト全体像
  - resources/scheme/   : 座組情報
  - resources/evaluation/ : 評価基準
- これらを読んで、AI が判断・実行に使える形に整理してほしい

成果物:
- .claude/skills/my-project/SKILL.md (マスター)
- .claude/skills/my-project/references/*.md (関心事別に分割)
- 元リソースで重要な情報は references/ に Markdown として転記
- PDF・画像・Excel などの生データは references には含めない
```

## キーワード「SSOT」を入れるとちゃんと作ってくれる

これ、地味に大事なノウハウなんですが、**プロンプトに "SSOT" や "Context Engineering" のような業界用語を入れると、AI がそれっぽく作ってくれます**。逆にこれらの言葉を使わずに「整理して」とだけ言うと、雑な構造になりがち。

「SSOT」は Context Engineering / Harness Engineering とセットでよく出てくるキーワードです。AI も学習データに大量にこの組み合わせを見てるので、ヒントになります。

## 一発では終わらない

正直に書くと、これは **1 回のプロンプトで終わりません**。

- 最初の生成: 「あ、`scheme.md` の構造もう少しこうしたい」
- 2 回目: 「`evaluation.md` のレベル定義、Excel 通りに表形式で書き直して」
- 3 回目: 「`schedule.md` を `overview.md` から分離して」

…という感じで、何度かやり取りしながら整えていきます。**初期コストはそれなりにかかる**。でもこれを 1 回ちゃんとやれば、後の作業が圧倒的に楽になる。

---

# Step 3: 整理されたスキルが完成、元リソースは用済み

数回のイテレーションで、こうなります：

```
my-project/
├── .claude/skills/my-project/
│   ├── SKILL.md
│   └── references/
│       ├── overview.md       ← AI が PDF やスライドから抽出して書いた
│       ├── scheme.md          ← AI がメンバー CSV と組織図から書いた
│       ├── evaluation.md      ← AI が Excel と業界スクショから書いた
│       ├── schedule.md
│       └── faq.md
└── resources/                  ← gitignore（もう用済み）
```

## 元リソースは gitignore して、コミットしない

PDF・Excel・スクショ・録音 — これらは：

- **重い**（リポジトリが肥大化）
- **AI に優しくない**（PDF は構造化されてないし、毎回 OCR が必要）
- **同じ情報が何回も書いてある**（ノイズ）
- **抽出後は用済み**（Markdown に落とせばそれが本物）

なので、`.gitignore` に `resources/` を追加して**ローカルだけに置く**。コミットするのは整理済みの `.claude/skills/` 配下だけ。

## ここから AI とのコミュニケーションが激変する

スキルが出来上がると、AI への指示がこうなります：

### Before（スキルなし時代）
```text
プロジェクト X についてヘルプして欲しい。背景としては、
うちの会社では半年に1回〇〇という取り組みをやっていて、
○○組には A 社と B 社が所属していて、責任者は…
（背景説明だけで 1000 トークン消費）

それで、Lv3 以上の組織を抽出して欲しいんだけど、Lv3 の定義は…
```

### After（スキルあり時代）
```text
my-project スキルを使って、Lv3 以上の組織一覧を出して。
```

これだけです。背景説明・用語定義・組織情報・評価基準、全部スキル内にあるので AI が自分で読みに行きます。**指示が業務語彙だけで完結する**。

副次的な効果として：

- **新メンバーへの説明** が `.claude/skills/my-project/` を見せて終わるようになる（AI 用と人間用が同じドキュメント）
- **意思決定の根拠** が AI から「`evaluation.md` のここの定義に基づいて Lv3 と判定しました」と返ってくる
- **属人化が消える**：「あの件は神谷さんに聞いて」が「あの件は my-project スキル見て」になる

# やってみて気づいた、設計のコツ

## コツ 1: スキルは「プロジェクトの本質的な軸」で分割する

`SKILL.md` 1 つに全部詰めるのではなく、references を **「自分の仕事の自然な分担軸」** で分ける。これは Step 1 のフォルダ構造とそのまま対応します。

私の場合は「座組 × 評価」の 2 軸でしたが、別のプロジェクトなら「フロントエンド × バックエンド × インフラ」かもしれないし、「営業 × カスタマーサクセス × プロダクト」かもしれない。**自分のプロジェクトを動かしている本質的な分担**を、フォルダ構造で表現する。

## コツ 2: スクリプトも references に置いていい

判定ロジックを Markdown で書くと曖昧になる場合は、**実行可能なスクリプト**（JS, Python, シェル）を `references/helpers/` に置く。AI に「このスクリプトを使って計算して」と指示できる。

```markdown
# evaluation.md

スコア算出:
- 14項目の平均値を取り、閾値テーブルでランク判定する
- 詳細は `helpers/score.js` を参照
```

これによって、AI が「コードを書く」ではなく「**用意されたコードを使う**」モードになるので、出力の再現性が上がります。

## コツ 3: description は呼び出しの命綱

YAML フロントマターの `description:` は、Claude Code が「このタスクにどのスキルを呼ぶか」を判断する**唯一のヒント**です。雑に書くと呼ばれない。

```yaml
# イマイチ
description: "プロジェクトを管理するスキル"

# ちゃんと呼ばれる
description: "プロジェクト X の全情報を統合管理するスキル。座組（メンバー・組織分担）、評価基準（Lv1-5定義）、スケジュール、FAQ を含む。「プロジェクトX」「A 組」「評価」「Lv3」等のキーワードで使用。"
```

入出力・キーワード・ユースケースを詰める。

## コツ 4: 「最初は綺麗じゃなくていい」

完璧な構造を最初から考えない。**雑に集める → AI に整理させる → 使ってみてズレを直す** のループを回す。

完璧主義になるとリソース集めが終わらない。30 分の議事録一個だけでも、まず `resources/` に放り込んで、AI に投げる。動かしながら整える。

# 「PM の仕事」が変わったという話

ここまで書いて自分でも改めて感じるのは、**PM の仕事の重心が完全に移った**ということです。

| 旧来の PM の仕事 | AI 時代の PM の仕事 |
|---|---|
| 情報を整理して人に伝える | AI が働きやすい形に情報を整理する |
| ステータスを集めてレポート化 | スキルを更新する |
| 「あれどこ？」に答える | スキルを充実させる（再発防止） |
| ドキュメントを書く | 素材を集めて AI に書かせる |
| メンバーオンボーディング | 「スキル見て」と渡す |

PM の本質的な価値は「**プロジェクトの全体像を理解していて、どこに何があるか分かっている**」ことだったはずです。それを AI が代行できるなら、PM はその一段上の仕事 — **「AI エージェントが働ける環境を設計する」 = Harness Engineering** に時間を使えます。

「自分の仕事が AI に奪われる」のではなく、「**自分の仕事を AI に渡せる形に再構築する**」のが、2026 年の PM の仕事の本体だと思っています。

---

# まとめ：3 ステップを貼っておきます

```
Step 1: リソースをかき集める
   └─ resources/ に PDF・スライド・スプシ・スクショ・議事録を全部突っ込む
   └─ プロジェクトの本質的な軸でフォルダを切っておく
   └─ gitignore で OK

Step 2: AI にスキルを作らせる（プランモード）
   └─ 「SSOT」「Context Engineering」をキーワードに目的を伝える
   └─ resources/ を読んで .claude/skills/ に整理させる
   └─ 1 回じゃ終わらないので何度かイテレーション

Step 3: 整理されたスキルが完成
   └─ SKILL.md(マスター) + references/*.md(関心事別)
   └─ スクリプトも references に置いていい
   └─ 元リソースは gitignore（用済み）
   └─ AI への指示が「スキル名 + 業務語彙」だけで完結するように
```

ここまでが**プロジェクトを AI と共に動かすための土台作り**です。

この上で、実際に AI に評価判定をさせたり、ダッシュボードを生成させたり、レポートを書かせたり…という具体的なタスクが乗ります。それは別記事で書きます。

似たような構築をしている PM の方、ぜひ話を聞かせてください。
